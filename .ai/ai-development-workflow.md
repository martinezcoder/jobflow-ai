# AI Development Workflow

## Purpose

This document describes the engineering workflow used in JobFlow AI.

The goal is not to maximize the amount of code generated by AI, but to maximize software quality while leveraging AI as a powerful engineering tool.

Every product decision, architectural decision and implementation follows a structured workflow designed to keep humans responsible for the outcome while using AI to accelerate development.

---

## Guiding Principles

### Humans make decisions

AI assists implementation, but product, architectural and technical decisions are made by humans.

### AI is a pair programmer

AI is treated as an engineering partner, not as an autonomous developer.

Its responsibility is to help implement well-defined requirements.

### One issue at a time

Only one GitHub Issue is implemented at a time.

Each issue has a clearly defined scope before implementation begins.

### Requirements before implementation

Every implementation starts from a GitHub Issue containing:

* User Story
* Acceptance Criteria
* Scope

Implementation never starts from an informal idea.

### Human review is mandatory

Every AI-generated change is reviewed before being committed.

No code is accepted without human validation.

### Continuous improvement

This workflow is expected to evolve as the project grows.

Engineering practices are regularly reviewed and refined.

# Team Responsibilities

## Product Owner

Responsible for:

* Product vision
* User value
* Backlog prioritization
* Sprint planning
* Acceptance Criteria
* Product decisions

## Tech Lead

Responsible for:

* Architecture
* Domain model
* Technical decisions
* Engineering standards
* Code review

## Developer

Responsible for:

* Understanding requirements
* Reviewing AI-generated code
* Testing
* Manual verification
* Creating commits
* Opening Pull Requests
* Final implementation quality

## AI Coding Assistant

Responsible for:

* Implementing approved requirements
* Suggesting improvements
* Writing tests
* Refactoring code
* Generating documentation when requested

The AI Coding Assistant never owns product decisions or architectural decisions.

# Development Lifecycle

Every issue follows the same engineering workflow.

```text
Sprint Planning
        ↓
Backlog Refinement
        ↓
GitHub Issue
        ↓
Implementation Prompt
        ↓
AI Implementation
        ↓
Human Review
        ↓
Pull Request
        ↓
CI Validation
        ↓
Merge
        ↓
AI Implementation Comment
        ↓
Sprint Review
```

---

# Sprint Planning

Each sprint begins with a planning session.

The objective is to:

* Define a Sprint Goal.
* Prioritize the backlog.
* Refine issues.
* Clarify product decisions.
* Identify risks.
* Agree on implementation order.

Only issues considered **Ready** should enter the sprint.

---

# Prompt Structure

Implementation prompts should contain only the information necessary for the current issue.

Long-term project knowledge belongs in the repository documentation rather than being duplicated in prompts.

Prompts should include:

* Issue identifier
* Issue title
* Issue type
* Context specific to the issue
* Acceptance Criteria
* Technical constraints
* Expected deliverables

---

# Human Review Process

Every implementation is manually reviewed before commit.

The review includes:

* Acceptance Criteria validation
* Product validation
* Architecture review
* Code quality review
* Automated tests
* Manual verification

If necessary, the implementation is refined before being accepted.

---

# AI Implementation Comments

Every completed GitHub Issue includes an AI Implementation comment documenting:

* AI model used
* Implementation prompt
* Human review summary

This creates a transparent and reproducible engineering history.

---

# Sprint Review

Each sprint ends with a Sprint Review.

The review documents:

* Sprint Goal
* Delivered functionality
* Product decisions
* Technical decisions
* Lessons learned
* Candidate backlog items for the next sprint

Sprint Reviews are versioned alongside the source code.

---

# Continuous Improvement

This document is intentionally treated as a living document.

Whenever the engineering process improves, this document should evolve accordingly.

The workflow is considered part of the project architecture and should receive the same level of care as the source code itself.
